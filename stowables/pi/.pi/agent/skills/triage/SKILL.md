# Triage

Move issues in Matt's **Local Markdown** tracker through a small state machine of triage roles.

Do not create remote tracker items. Do not use remote tracker CLIs. Ignore GitHub, GitLab, Linear, Jira, and any other remote issue tracker.

Every comment or issue text added during triage should start with this disclaimer:

```markdown
> *This was generated by AI during triage.*
```

## Reference docs

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — how to write durable agent briefs
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — how the `.out-of-scope/` knowledge base works

## Local tracker

- One feature per directory: `.scratch/<feature-slug>/`
- The spec is `.scratch/<feature-slug>/PRD.md`
- Issues/tickets are `.scratch/<feature-slug>/issues/<NN>-<slug>.md`, numbered from `01`
- State is recorded as a `Status:` line near the top
- Comments append under `## Comments`

## Roles

Two **category** roles:

- `bug` — something is broken
- `enhancement` — new feature or improvement

Five **state** roles:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter
- `ready-for-agent` — fully specified, ready for an AFK agent
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Every triaged issue should carry exactly one state via `Status:`. Put category in a `Category:` line if the issue does not already have one.

## Invocation

The maintainer invokes `/skill:triage` and describes what they want in natural language. Examples:

- "Show me anything that needs my attention"
- "Let's look at `.scratch/foo/issues/03-bar.md`"
- "Move issue 03 to ready-for-agent"
- "What's ready for agents to pick up?"

## Show what needs attention

Scan `.scratch/*/issues/*.md` and present three buckets, oldest first:

1. Missing `Status:` or `Status: needs-triage`
2. `Status: needs-info` with comments after the last triage note
3. `Status: ready-for-agent` but missing a durable agent brief

Show counts and a one-line summary per item. Let the maintainer pick.

## Triage a specific issue

1. **Gather context.** Read the full issue body and comments. Explore the codebase using the project's domain glossary, respecting ADRs in the area. Run two checks: (a) **redundancy** — search for an existing implementation of the requested behavior by domain concept; if found, recommend `wontfix` as already implemented. (b) **prior rejection** — read `.out-of-scope/*.md` and surface any that resembles this request.

2. **Recommend.** Tell the maintainer your category and state recommendation with reasoning, plus a brief codebase summary relevant to the request. Wait for direction.

3. **Verify the claim.** For a bug, reproduce it from the reporter's steps. For an enhancement, verify the current behavior and where the change would attach. Report confirmed, failed, or insufficient detail.

4. **Grill if needed.** If the request needs fleshing out, use `/skill:grilling` and `/skill:domain-modeling` together — one question at a time, sharpening domain terms and updating `CONTEXT.md`/ADRs inline as decisions land.

5. **Apply the outcome:**
   - `ready-for-agent` — append an agent brief comment ([AGENT-BRIEF.md](AGENT-BRIEF.md)) and set `Status: ready-for-agent`.
   - `ready-for-human` — append the same structure, but note why it can't be delegated.
   - `needs-info` — append triage notes using the template below and set `Status: needs-info`.
   - `wontfix` — set `Status: wontfix`; for rejected enhancements, write to `.out-of-scope/` and link it from the comment.
   - `needs-triage` — set the status and optionally append partial progress.

## Quick state override

If the maintainer says "move issue 03 to ready-for-agent", trust them and update the local file directly after confirming the exact path and edit. Skip grilling. If moving to `ready-for-agent` without a grilling session, ask whether they want to write an agent brief.

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you:**

- question 1
- question 2
```

Capture everything resolved during grilling under "established so far" so the work isn't lost. Questions must be specific and actionable, not "please provide more info".
